================================================================================
ANOMALY DETECTION FOR BANK STATEMENTS - APPROACH & IMPLEMENTATION SUMMARY
================================================================================

PROJECT OBJECTIVE:
Detect suspicious or fraudulent transactions in bank statements by identifying
patterns that deviate from normal spending behavior.

================================================================================
DETECTION METHODOLOGY
================================================================================

The anomaly detection system uses FIVE complementary detection methods:

1. STATISTICAL OUTLIER DETECTION (Amount-Based)
   - Method: Z-Score Analysis
   - Identifies: Transactions with unusually high or low amounts
   - Threshold: > 2.5 standard deviations from historical mean
   - Why it works: Fraud typically involves much larger/smaller amounts than normal
   - Example: User spends avg $50/day → $5,000 charge flagged as outlier

2. MERCHANT PATTERN ANALYSIS
   - Method: Vendor history tracking with frequency analysis
   - Identifies:
     * Transactions to NEW merchants (never seen before)
     * RARE merchants (only 1-2 prior transactions)
   - Why it works: Fraud often uses unfamiliar vendors
   - Example: User always shops at Amazon & Target → charge to "CryptoExchange" flagged

3. TIME-BASED ANOMALY DETECTION
   - Method: Hour-of-day frequency analysis
   - Identifies: Transactions at unusual hours (low historical frequency)
   - Threshold: Transactions in hours <20% of average frequency
   - Why it works: Most people shop during normal hours; fraud happens 3am
   - Example: No transactions before 7am historically → 3:22am charge flagged

4. DUPLICATE DETECTION
   - Method: Exact-match and near-match analysis
   - Identifies:
     * Identical amounts to same merchant within 24 hours
     * Processing errors or repeated fraudulent charges
   - Why it works: Duplicates are rare in legitimate statements
   - Example: Two identical $42.50 Target charges same minute = likely error/fraud

5. FREQUENCY & CONCENTRATION ANALYSIS
   - Method: Spending pattern frequency detection
   - Identifies: Unusual concentration of high-value transactions
   - Threshold: >30% of transactions are 2x average amount
   - Why it works: Legitimate spending is distributed; fraud is concentrated
   - Example: 5 of 16 transactions are >$1000 (historically rare)

================================================================================
BASELINE ESTABLISHMENT
================================================================================

The system requires historical data (3+ months recommended) to establish:
- Average spending amounts and standard deviation
- List of trusted merchants with usage frequency
- Typical transaction hours
- Normal spending patterns

This baseline is then used to compare current month's transactions against.

================================================================================
IMPLEMENTATION DETAILS
================================================================================

PYTHON CLASS: TransactionAnomalyDetector
Location: anomaly_detection.py

Key Methods:
- __init__(): Initialize with historical transaction baseline
- detect_anomalies(): Run all detection algorithms on current transactions
- _detect_statistical_outliers(): Z-score based amount detection
- _detect_merchant_anomalies(): New/rare merchant detection
- _detect_time_anomalies(): Unusual hour detection
- _detect_duplicates(): Duplicate transaction detection
- _detect_frequency_anomalies(): Spending concentration detection
- get_anomaly_report(): Generate structured JSON report
- export_json(): Save report to file

SEVERITY SCORING:
- Z-score method: severity = min(z_score / 5.0, 1.0)
- New merchant: severity = 0.6
- Rare merchant: severity = 0.4
- Time anomaly: severity = 0.35
- Duplicate: severity = 0.8
- Frequency anomaly: severity = 0.5

Range: 0.0 (minimal concern) to 1.0 (critical/immediate action)

HOW TO USE
================================================================================

BASIC USAGE:
```python
from anomaly_detection import TransactionAnomalyDetector

# Initialize with historical transactions
detector = TransactionAnomalyDetector(historical_transactions=feb_transactions)

# Analyze current month
anomalies = detector.detect_anomalies(march_transactions)

# Generate report
report = detector.get_anomaly_report()
detector.export_json('march_anomalies.json')
```

EXPECTED INPUT FORMAT:
Each transaction should be a dictionary with:
{
    'date': '2026-03-12 11:00:00',  # ISO format or similar
    'merchant': 'Vendor Name',
    'amount': '1234.50',            # String or float
    'type': 'debit',                # 'debit' or 'credit'
    'description': 'Purchase description'
}

EXPECTED OUTPUT:
JSON report containing:
- Total transactions analyzed
- Total anomalies found (by type)
- Detailed anomaly records with:
  * Transaction details
  * Anomaly type
  * Severity score (0.0-1.0)
  * Reason/explanation
  * Recommended action

================================================================================
FRAUD DETECTION BEST PRACTICES
================================================================================

1. ESTABLISH BASELINE
   - Collect 3+ months of historical transactions
   - Ensure period is representative (no vacation/special events)
   - Recalibrate baseline quarterly

2. MONITOR REGULARLY
   - Run detection algorithm on each statement
   - Review at least weekly during active monitoring
   - Check within 24 hours of dispute deadline

3. VERIFY ALERTS
   - Not all anomalies are fraud (legitimate changes happen)
   - Contact merchants for unusual charges
   - Keep receipts for verification
   - Report quickly to bank (usually 60-day window)

4. TAKE ACTION
   - Dispute unauthorized charges immediately
   - Update passwords if vendor was compromised
   - Monitor credit reports for identity theft
   - Consider card replacement for large fraud

5. REFINE RULES
   - Adjust thresholds as spending patterns change
   - Account for seasonal variations (holidays, travel)
   - Learn from false positives to improve accuracy

================================================================================
LIMITATIONS & CONSIDERATIONS
================================================================================

1. REQUIRES HISTORICAL DATA
   - New accounts with <3 months history have reduced effectiveness
   - Major life changes (relocation, job change) require recalibration

2. CHANGING SPENDING PATTERNS
   - Legitimate increases (home purchase, medical) can trigger alerts
   - Seasonal variations need consideration
   - Updates to detector thresholds may be needed

3. FALSE POSITIVES POSSIBLE
   - Valid large purchases may be flagged
   - New legitimate merchants will trigger alerts
   - Manual review required to validate

4. NOT A GUARANTEE
   - System flags suspicious patterns, not definitive fraud
   - Some fraud may not match statistical patterns
   - Multi-factor verification always recommended

5. MERCHANT NAME VARIATIONS
   - Same vendor may appear as different names
   - Merchant normalization helps but not perfect
   - Manual verification sometimes needed
