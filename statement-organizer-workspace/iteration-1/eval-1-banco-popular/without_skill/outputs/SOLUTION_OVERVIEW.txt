================================================================================
DESIGN DECISIONS & RATIONALE
================================================================================

1. MULTI-FACTOR DETECTION
   Why: Single factor (headers alone) could match other Spanish banks
   How: Combine headers + patterns + dates for 85%+ confidence
   Benefit: Accurate identification with low false positive rate

2. SPANISH KEYWORD MATCHING
   Why: Transaction descriptions are in Spanish with bank-specific terms
   How: Regex patterns match known Banco Popular keywords
   Benefit: Automatic categorization without manual configuration

3. FLEXIBLE NUMERIC PARSING
   Why: Banco Popular uses "-" for zero amounts, not just "0" or empty
   How: Replace "-" with 0.0, handle empty as 0.0
   Benefit: Robust parsing of non-standard formats

4. BALANCE VERIFICATION
   Why: Ensures data integrity and catches parsing errors
   How: Verify each row: balance = previous + deposit - withdrawal
   Benefit: Catch data corruption or format mismatches early

5. NON-BLOCKING ERROR HANDLING
   Why: Bad rows shouldn't stop entire import
   How: Skip bad rows with warning, continue processing
   Benefit: Partial statements can still be processed

6. STRUCTURED OUTPUT (JSON)
   Why: Easy to parse, version control friendly, tool-independent
   How: Export transactions as JSON with full metadata
   Benefit: Can feed to other tools, databases, or analyses
